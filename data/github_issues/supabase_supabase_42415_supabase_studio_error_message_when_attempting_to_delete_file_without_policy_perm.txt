TITLE:
Supabase Studio error message when attempting to delete file without policy permission

ISSUE BODY:
# Chore

## Describe the chore

It would be nice if the storage module gave an error message like "User does not have permission to delete files. Please update bucket policy." when attempting to delete a file. Right now it just acts as if it was successfully deleted, when it really wasn't.

## Additional context

This is the behavior I experienced when trying this with the default Supabase Docker Compose setup (Supabase Studio version 2025.11.26).

Dug into this one. The silent success is structural: Studio shows the success toast regardless of how many objects were actually removed.

 The delete goes through deleteBucketObject (https://github.com/supabase/supabase/blob/e9888de2c0dc60144cbd045295b2deb691e1e02c/apps/studio/data/storage/bucket-object-delete-mutation.ts#L12-L33), which only throws on a non-2xx (the error runs through handleError, typed never), and deleteFiles fires Successfully deleted N file(s) (https://github.com/supabase/supabase/blob/e9888de2c0dc60144cbd045295b2deb691e1e02c/apps/studio/state/storage-explorer.tsx#L1483-L1488) on anything that does not throw. It never checks what the response says was removed:

 - Hosted: StorageObjectsController_deleteObjects (https://github.com/supabase/supabase/blob/e9888de2c0dc60144cbd045295b2deb691e1e02c/packages/api-types/types/platform.d.ts#L29480-L29485) is typed 200: { content?: never }, so there is no body to inspect.
 - Self-hosted: the route (https://github.com/supabase/supabase/blob/e9888de2c0dc60144cbd045295b2deb691e1e02c/apps/studio/pages/api/platform/storage/%5Bref%5D/buckets/%5Bid%5D/objects/index.ts#L20-L30) returns the remove() result on 200, which is the list of objects actually deleted with a null error, so a path that matched nothing comes back as { data: [], error: null } and still hits the success toast. (This route uses the service_role key, so RLS is bypassed here; the empty result is a no-match case rather than a policy denial.)

 Making delete honest means comparing the requested paths against what was actually removed. That set is available self-hosted; on hosted the endpoint would need to return it (or 403 when it removes nothing). Flagging so whoever picks this up scopes it right rather than just swapping the toast.
