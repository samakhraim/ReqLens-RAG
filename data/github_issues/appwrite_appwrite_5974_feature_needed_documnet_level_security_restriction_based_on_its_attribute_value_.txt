Hi @rahul3v 
Thanks for sharing your thoughts.
To address the specific example you provided—where users should only see posts related to their organization—it's worth noting that this can already be achieved with Appwrite's current structure:

Given your relationships, where:
1. Users have a one-to-many relation with posts.
2. Organizations have a one-to-many relation with users.

Here's a suggested approach:

**Using Teams**:
- Create a separate team for each organization.
- When creating the posts, assign read permissions specifically to that team.
- Therefore, when users fetch posts, they will only be able to retrieve posts that are visible to members of their specific organization's team.

Since you mentioned Cloud, our Cloud uses 1.1.2 version of Appwrite and hence you might not be able to use Relationships. But, we are working on updating it to the latest version.

That said, we always appreciate feedback. Thank you for your understanding and for being part of our community!

Body:
Suppose I want to allow the posts to be viewed based on these conditions:
- Post have to be marked as published.
- The post can be viewed without any subscription for first 1000 views (can be modified by author) only if the user follow the post author
- But if user is a paid user then they can view any post (1st restrictions still apply)
- If user and author belongs to same organisation then the user can view the post no matter if it is published or not and they not have to be a paid user.

To implement this we have to use multiple team and also to allow free access for first 1000 views for users who follow the author we have to use appwrite functions.

Where as if we implement permission based on attribute value and it's relationship's attribute then it will be much much easier to implement.

Here is how it looks for [PocketBase](https://pocketbase.io):
```
(status = "published" &&
((views <= allowedFreeViews && @request.auth.follows ?= author.id) ||
@request.auth.isPaid = true)) ||
@request.auth.organisation.id = author.organisation.id
```
For PocketBase implementation view [this page](https://pocketbase.io/docs/api-rules-and-filters/).

Although for giving access to paid user and organisation member "Team" feature is a good use case but there have many use cases where permission based on attribute value and it's relationship's attribute will be easier to implement without writting custom code using functions.
